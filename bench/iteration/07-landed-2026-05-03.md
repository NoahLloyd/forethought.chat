# Landed iteration: 2026-05-03 (BENCHMARK_VERSION 0.3.0)

Snapshot of what shipped from the proposals in `03-` and `04-`. Earlier docs
(`01-` through `06-`) are still valid as design context; this one says what
is NOW IN THE BENCH and what changed in scoring shape.

## What shipped

### A3 — LLM-judge for numeric correctness

`bench/forethought_bench/scoring/numeric_judge.py` introduces
`score_numeric_judge(answer, target, judge)`. The judge is given the target
value, unit, and tolerance and returns CORRECT (1.0) / PARTIAL (0.5) /
INCORRECT (0.0). Wired into `claim_recall.py` in place of the regex-based
`score_numeric`.

Why a judge instead of extending the regex: word-form prose like
`eightfold`, `two-thirds`, `a tenfold increase`, `a factor of eight` is
unbounded surface area. The regex was already on its second patch and would
need a third, fourth, etc. The judge handles the whole space natively.

The regex (`numeric_tolerance.score_numeric`) is still exported and tested;
keeping it means there is a known-good extractor for offline tests and any
future use that needs determinism.

### A1 — Claim-anchored citation refinement

`bench/forethought_bench/scoring/claim_anchoring.py` introduces
`refine_citation_claims(output, judge)`. It re-runs the agent's prose
through an LLM extractor that splits each marker-bearing sentence into the
smallest claim each marker backs, then rewrites the `Citation.supports`
field on each citation. Runs BEFORE `check_all_citations` in every track
scorer.

Failure mode this fixes: a sentence "Claim X is true and Y has 50% chance
[1] [2]" used to give both citations `supports="Claim X is true and Y has
50% chance"`. The support judge then asked "does chunk [1] support 'X is
true and Y has 50% chance'?" and got UNSUPPORTIVE because chunk [1] only
backs the X clause. After A1, citation [1] is asked about "X is true" and
[2] about "Y has 50% chance".

When the agent's citation list is out of sync with the deterministic walk
(custom agent that emits its own citations), refinement is a no-op — the
deterministic supports text is preserved.

### A2 — Per-document holistic answer-support grader

`bench/forethought_bench/scoring/answer_support.py` introduces
`score_answer_support(output, corpus, judge)`. It builds an evidence block
from all distinct cited URLs (deduped, capped at 24K chars), strips `[N]`
markers from the answer, and asks: "given the evidence block as the only
allowed source, are there factual claims in the answer that are NOT
supported?" The judge returns an `unsupported_claims` list; the score is
1.0 with zero unsupported, dropping to 0.2 with three or more.

This catches two failures the per-claim grader can't see:
- A claim is jointly supported by 2+ chunks but no single chunk on its own.
  Per-claim says PARTIAL/UNSUPPORTIVE for both citations; per-document says
  fully supported.
- A claim in the answer that no cited source supports. Per-claim has no
  marker to walk from, so it can't see this; per-document flags it.

When the answer has no resolvable cited evidence, the grader returns
`score=0.5` (neutral) and sets `judge_ran=False`.

### Boundary track removed

Reason in steering note: the gate decides routing before the librarian sees
the query. Testing the librarian for refusal was testing the wrong
abstraction. The librarian should always try its best once it gets a query.

Deleted:
- `bench/forethought_bench/gate/`
- `bench/items/gate/`
- `bench/scripts/run_gate.sh`
- `BOUNDARY` from `TrackName` enum
- `expected_behavior` and `boundary_subtype` from `Item`
- Boundary-specific render path in `render_report.py`
- Old gate logs (`logs/full_gate`, `logs/smoke_gate`)

## Wall-time delta

Each track now makes 2 extra judge calls per item: one for claim anchoring,
one for answer support. On Haiku 4.5 at ~1.5s/call, that's ~3s per item ×
18 smoke items = ~55s extra wall on a smoke run. Numeric judge replaces the
free regex call, adding ~1.5s × 5 = ~8s on claim_recall. Total smoke run
goes from ~5-7 min to roughly ~7-9 min.
